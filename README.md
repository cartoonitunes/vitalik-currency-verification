Exact bytecode reproduction of the token contract deployed by Vitalik Buterin on November 12, 2015.

## Result: Exact Match

The contract was compiled with the **Serpent compiler**, not Solidity.

| Detail | Value |

| Language | Serpent |

## Why Serpent, Not Solidity?

The `ethereum/dapp-bin` repository contained both `currency.sol` (Solidity) and `currency.se` (Serpent) implementations of the standardized currency API. The deployed contract matches the Serpent version.

This makes sense historically: Vitalik created Serpent, and in November 2015 it was still the language he used for contract deployment. Solidity was maturing but had not yet displaced Serpent for all use cases.

## How We Found It

We initially tried to verify this contract by compiling `currency.sol` with every available Solidity compiler version. None matched.

Three clues pointed to Serpent:

1. **Constructor pattern:** The on-chain init code starts with `6000603f53` (`MSTORE8`-based memory initialization), which is generated by Serpent's `finalize()` function. Solidity uses `60606040525b` (the free memory pointer pattern).

2. **Memory allocation:** The runtime code uses `MSIZE, SWAP1, MSIZE, ADD` for memory allocation - Serpent's `alloc()` pattern, not found in any Solidity version.

3. **Function name differences:** Two on-chain selectors didn't match the Solidity source:

| On-chain selector | Serpent function | Solidity equivalent |
|---|---|---|
| `0x181670e6` | `isApprovedOnceFor(address,address)` | `isApprovedOnce(address,address)` |
| `0x15770d99` | `disapprove(address)` | `unapprove(address)` |

## Verification

To reproduce (requires Serpent compiler built from commit `f0b4128`):

```bash
git clone https://github.com/ethereum/serpent
cd serpent && git checkout f0b4128
make
./serpent compile currency.se
```

The output should match `onchain-bytecode.txt` exactly.

## What Was Tried (and failed)

Before finding the Serpent match, we tested:

- `soljson` JavaScript releases v0.1.1 through v0.3.6 (stable + nightlies, Sep-Dec 2015)
- Native C++ `solc` builds from `webthree-umbrella` (Oct-Nov 2015 commits)
- Native C++ `solc` from `cpp-ethereum` monorepo (pre-split era, Feb-Jun 2015)
- Optimizer on and off for every version
- Ethereum PPA packages and Mist release bundles

All Solidity compilers produced different bytecode patterns (free memory pointer init, different code size, wrong function selectors for two functions).

## Etherscan Verification

Etherscan does not currently support Serpent as a verification language. Manual verification has been requested, providing:

- The exact source code (`currency.se`)
- The compiler version (ethereum/serpent commit `f0b4128`)
- Byte-for-byte proof of bytecode match
